4. **"Remote gateways must use HTTPS/WSS"** then **"Discovered gateway is missing TLS"** — the iOS app has two separate connection paths: an auto-discovered Bonjour/LAN entry (plain, unencrypted) and the manually-paired Tailscale Serve entry (`wss://`). Dave was hitting the auto-discovered one; switched to the paired entry.

**Lesson worth remembering:** always check for orphaned gateway processes (`ps` + `launchctl list | grep ai.openclaw.gateway`) before assuming a config change didn't take effect. `openclaw gateway restart` can silently fail to replace a stuck process without erroring loudly every time.

## ElevenLabs TTS — two outages, then fixed properly

Dave wanted ElevenLabs voice set up. This went sideways before landing clean — full trail below since it's a good case study in what NOT to do live on a production gateway.

**What went wrong (twice):** wired `messages.tts.providers.elevenlabs.apiKey` as a SecretRef pointing at a 1Password-backed exec provider, restarted the gateway, and it **crash-looped** both times with `messages.tts.providers.elevenlabs.apiKey resolved to a non-string or empty value` — meaning the *whole gateway* (iMessage, Gmail, everything) was down each time until reverted. `--dry-run --allow-exec` passed cleanly both times despite this, which turned out to be a red herring — it does not accurately simulate the real boot-time secrets resolution path.

**Actual root cause (confirmed by direct reproduction, not guessing):** the exec-provider secrets script called `op read`, but the exec provider runs commands with a **fully stripped environment** by default — no `HOME`. Without `HOME`, `op` can't find its config directory, fails, and the wrapper script's `$(...)` capture silently swallowed the stderr error, producing a syntactically-valid-but-empty JSON envelope (`{"values":{"value":""}}`) — which matches doctor's exact wording ("non-string **or empty**"). Reproduced directly: `env -i <script>` fails identically; `env -i HOME=... <script>` succeeds. This was missed because an earlier draft of the provider config had `--provider-pass-env HOME` set, but it got dropped when the config was rebuilt mid-troubleshooting.

**Fix applied:**
1. Added `--provider-pass-env HOME` back to the `secrets.providers.op-elevenlabs` registration.
2. Hardened **all four** `op-*.sh` scripts (not just elevenlabs) to check `op`'s exit code and emit a proper `errors` field with the real stderr message instead of silently emitting an empty value on failure — this same latent bug existed in the other three scripts too, just hadn't bitten because they aren't wired into gateway-boot-critical config.
3. Verified via direct `env -i` / `env -i HOME=...` reproduction (not `--dry-run`, which doesn't catch this class of bug) before touching the live gateway again.
4. Applied to production, restarted once, gateway came up clean.
5. Confirmed with `openclaw infer tts convert` — the pipeline works end-to-end (real authenticated call reached ElevenLabs). Only remaining blocker: ElevenLabs itself rejects the example "library" voice ID on Dave's account tier (`402: Free users cannot use library voices via the API`) — a legitimate account/plan limitation, not a bug. Dave's decision whether to upgrade ElevenLabs or use a voice type available on the free tier (e.g. a cloned voice).

**Process lesson for next time:** don't wire a new SecretRef directly into gateway-boot-required config and restart live to test it. `--dry-run --allow-exec` is not sufficient — it validated successfully both times the real thing failed. Test the actual script standalone with `env -i` (and `env -i HOME=...`) to see exactly what environment it needs *before* registering it as a provider.
